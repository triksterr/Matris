Файл контекста проекта для Codex

1. Общие правила
- Язык кода: C++17
- Язык комментариев в коде: русский
- Язык диалога и объяснений: русский
- Кодировка текстовых файлов: UTF-8
- Для навигационных комментариев использовать ключевое слово ` Link: ` (с заглавной L и пробелами до и после ссылки, по одной ссылке на строку), совместимое с плагином MS VS:
  ` Link:..\cons_test\cons_test.cpp ` - ссылка на файл
  ` Link:..\cons_test\cons_test.cpp#L14 ` - ссылка на строку 14 в файле
  ` Link:..\cons_test\cons_test.cpp:"TEST #t08 - задача тестирования №8" ` - переход на строку с указанным текстом
- Все изменения пояснять подробно: что изменено, зачем, где именно и как это проверить. Формат отчёта об изменениях: файл/функция -> что изменено -> как проверить
- Комментарии писать подробно везде, где логика неочевидна
- Любое обнаруженное изменение файла считать изменением пользователя по умолчанию.
- Можно уточнять у пользователя причину внесённых им изменений и при необходимости объяснять, почему эти изменения лучше не вносить.
- Отменять, откатывать или перезаписывать изменения пользователя можно только после его явного подтверждения.

3. Текущая цель этапа
Сделать минимально рабочий прототип консольной версии:
- проанализировать все существующие классы и методы в cons_test на предмет корректности и ошибок
- завершить незавершенные функции, включая поворот фигуры и проверку возможности поворота
- реализовать падение фигуры, фиксацию фигуры в слоях и применение правил раунда
- реализовать минимальный базовый функционал только для консоли и клавиатуры (без файлового ввода-вывода и без звука на данном этапе)
- покрыть базовый функционал автоматическими тестами (GoogleTest)
- разработать методику ручного тестирования и выполнить ручное тестирование

4. Критерии готовности текущего этапа
- Базовая сборка этапа: Visual Studio, конфигурация Debug|x64, без ручных правок
- Сдача этапа: локальный запуск из VS + запуск в консоли без VS + минимальный smoke-тест
- Целевой ритм игрового цикла: 60 FPS (если невозможно на конкретной системе/железе - выносить на обсуждение)
- Небольшой дрейф времени шага допускается, если система/железо не тянут целевой режим
- Базовый размер поля: 20x10, допустимы вариации размеров в диапазоне +-3
- Шаг падения фигуры: 1 секунда (параметр правила раунда, может изменяться от раунда к раунду)
- Lock delay после касания снизу: 1 секунда (параметр правила раунда, может изменяться от раунда к раунду)
- После касания снизу в течение lock delay разрешено только движение влево-вправо
- Условие Game Over: новая фигура при появлении пересекается с кубиками слоев
- Задача считается завершенной только после: код + сборка + автотесты + ручная проверка + запись результата в /prj/Codex_todo.txt

5. Definition of Done по задачам этапа
- Анализ классов и методов (DoD):
  выполнен анализ всех классов и методов в cons_test; найденные ошибки/риски оформлены в /prj/class_method_analysis_report.md и заведены в /prj/Codex_todo.txt как задачи с приоритетом.
- Поворот фигуры и проверка возможности поворота (DoD):
  незавершенные функции дописаны; поворот выполняется вокруг заданного центрального кубика; при невозможности поворота поворот отменяется; проверка учитывает границы стакана и нижние слои.
- Падение и фиксация фигуры (DoD):
  фигура падает с шагом 1 секунда; после касания снизу действует lock delay 1 секунда с разрешением только движения влево-вправо; после этого фигура целиком переходит в слои.
- Правила раунда тестового этапа (DoD):
  правило 1 (гравитация): каждый кубик слоев, под которым нет опоры, проваливается до касания с кубиком слоев или дном;
  правило 2 (уничтожение): ищется первая пара кубиков с суммой 10 по порядку проверки: вертикаль, слева, справа, сверху; удаляется только первая найденная пара;
  после любого проваливания и/или уничтожения цикл "гравитация -> уничтожение" повторяется до стабилизации состояния.
- Минимальный базовый функционал этапа (DoD):
  реализованы и проверены только консоль и клавиатура; файловый ввод-вывод и звук в текущий этап не входят.
- Автотесты (DoD):
  используется GoogleTest; покрыты все публичные методы ключевых классов и ключевые сценарии логики раунда.
- Ручное тестирование (DoD):
  разработана методика ручного тестирования; ручное тестирование выполнено; результаты зафиксированы в /prj/Codex_todo.txt.
- Кроссплатформенное ограничение (DoD):
  в src_base не использовать Windows-специфику; платформенные детали размещать в соответствующих платформенных слоях.

6. Обязательный ручной smoke-чеклист этапа
- 1) Собрать Debug|x64 без ручных правок проекта.
- 2) Запустить из Visual Studio, проверить старт без критических ошибок.
- 3) Запустить бинарник из консоли без Visual Studio.
- 4) Проверить клавиши: влево/вправо/вниз/поворот/сброс/пауза/выход в меню.
- 5) Проверить корректность перерисовки: минимум без артефактов и накопления визуального мусора; целевой режим - стабильное время кадра.
- 6) Проверить поворот вокруг центрального кубика и запрет поворота при невозможности (границы/слои).
- 7) Проверить падение фигуры с шагом 1 сек и lock delay 1 сек с разрешением только движения влево-вправо.
- 8) Проверить переход фигуры в слои после lock delay.
- 9) Проверить правило гравитации слоев и правило уничтожения пары (сумма 10) с порядком проверки: вертикаль, слева, справа, сверху.
- 10) Проверить циклическое применение правил раунда до стабилизации.
- 11) Проверить условие Game Over при пересечении новой фигуры со слоями при появлении.
- 12) Проверить выполнение автотестов GoogleTest.
- 13) Зафиксировать результат выполнения в /prj/Codex_todo.txt.

7. Архитектурные правила
- Общая игровая логика должна по возможности жить отдельно от платформенного кода
- Платформозависимые части изолировать: консоль, клавиатура, звук, позже оконный режим
- Если логика потенциально понадобится на Linux/Android, не привязывать её к WinAPI без необходимости
- Новые модули делать небольшими и с одной понятной ответственностью

8. Правила кода
- Без лишних глобальных переменных
- Предпочитать понятный код сложным оптимизациям
- Не менять чужие или архивные части проекта без явной необходимости
- При добавлении новых сущностей кратко фиксировать их назначение в этом файле
- Проект должен собираться и работать на всех заявленных платформах
- Исключения C++: try/catch запрещены, обработка ошибок через bool/enum/коды ошибок
- RTTI (dynamic_cast, typeid): полный запрет до появления реальной необходимости
- Сырые указатели: использовать существующий подход проекта, новые решения принимать по ситуации при реальной необходимости
- Механизмы управления памятью (включая shared_ptr/unique_ptr/new/delete): использовать существующий подход проекта, изменения по необходимости и по ситуации
- C-style массивы и строки (char*, sprintf): допускаются
- Стиль имён: PascalCase
- Имена классов и структур: PascalCase
- Префиксы имён (m_, g_, s_): не использовать
- Имена констант: по аналогии с существующими в проекте
- Имена файлов исходников и заголовков: snake_case.h/.cpp
- STL-контейнеры: стандартные контейнеры использовать по необходимости
- "Экзотические" контейнеры и нестандартные подходы: только по явному запросу
- Строго использовать возможности C++17, без аналогов из более новых стандартов
- Передача контейнеров в функции: по аналогии с существующим кодом проекта
- range-for: использовать по необходимости и по запросу
- auto: использовать по аналогии с существующим кодом проекта
- Макросы: разрешены include guards, платформенные #if и compile-time флаги; функциональные макросы запрещены
- Ссылки и указатели в параметрах функций: решение по месту (вариант C), в соответствии с ситуацией
- assert: обязателен расширенно для инвариантов и невозможных состояний в src_base; в платформенных адаптерах использовать для проверок предпосылок
- Логирование: обязательно для ошибок I/O и ошибок платформенных API в src_cons/src_win/src_andr; использовать уровни Error/Warn/Info (диагностический минимум), в игровом и рендер-цикле без спама каждый кадр
