# Global Rules

## Authority

High authority operational rules.

При конфликте:
- source code authoritative для implementation;
- данный файл authoritative для workflow/code-style/governance rules.

---

## Общие правила

- Язык кода: C++17
- Язык комментариев в коде: русский
- Язык диалога и объяснений в чате: русский
- Кодировка текстовых файлов: UTF-8
- Для навигационных комментариев использовать ключевое слово ` Link: ` (с заглавной L и пробелами до и после ссылки, по одной ссылке на строку), совместимое с плагином MS VS:
  ` Link:..\cons_test\cons_test.cpp ` - ссылка на файл
  ` Link:..\cons_test\cons_test.cpp#L14 ` - ссылка на строку 14 в файле
  ` Link:..\cons_test\cons_test.cpp:"TEST #t08 - задача тестирования №8" ` - переход на строку с указанным текстом
- Все изменения пояснять подробно: что изменено, зачем, где именно и как это проверить. Формат отчёта об изменениях: файл/функция -> что изменено -> как проверить
- Комментарии писать подробно везде, где логика неочевидна
- Любое обнаруженное изменение файла считать изменением пользователя по умолчанию.
- Можно уточнять у пользователя причину внесённых им изменений и при необходимости объяснять, почему эти изменения лучше не вносить.
- Отменять, откатывать или перезаписывать изменения пользователя можно только после его явного подтверждения.
- Архитектурные изменения требуют отдельного согласования USER
- Scope expansion требует явного подтверждения USER
- Unrelated refactoring запрещен без отдельного approval

### Context Discipline

- Использовать bounded-context execution
- Не загружать нерелевантные subsystem/task files
- Избегать unnecessary context expansion
- Предпочитать minimal mutation scope
- Не выполнять speculative redesign без запроса USER

## Архитектурные правила

- Общая игровая логика должна по возможности жить отдельно от платформенного кода
- Платформозависимые части изолировать: консоль, клавиатура, звук, позже оконный режим
- Если логика потенциально понадобится на Linux/Android, не привязывать её к WinAPI без необходимости
- Новые модули делать небольшими и с одной понятной ответственностью

---

## Правила кода

- Без лишних глобальных переменных
- Предпочитать понятный код сложным оптимизациям
- Не менять чужие или архивные части проекта без явной необходимости
- При добавлении новых subsystem-level сущностей обновлять соответствующие subsystem docs при необходимости
- Архитектура должна учитывать заявленные future platforms
- Platform-independent logic не должна unnecessarily зависеть от WinAPI
- Исключения C++: try/catch запрещены, обработка ошибок через bool/enum/коды ошибок
- RTTI (dynamic_cast, typeid): полный запрет до появления реальной необходимости
- Сырые указатели: использовать существующий подход проекта, новые решения принимать по ситуации при реальной необходимости
- Механизмы управления памятью (включая shared_ptr/unique_ptr/new/delete): использовать существующий подход проекта, изменения по необходимости и по ситуации
- C-style массивы и строки (char*, sprintf): допускаются
- Стиль имён: PascalCase
- Имена классов и структур: PascalCase
- Префиксы имён (m_, g_, s_): не использовать
- Имена констант: по аналогии с существующими в проекте
- Имена файлов исходников и заголовков: snake_case.h/.cpp
- STL-контейнеры: стандартные контейнеры использовать по необходимости
- "Экзотические" контейнеры и нестандартные подходы: только по явному запросу
- Строго использовать возможности C++17, без аналогов из более новых стандартов
- Передача контейнеров в функции: по аналогии с существующим кодом проекта
- range-for: использовать по необходимости и по запросу
- auto: использовать по аналогии с существующим кодом проекта
- Макросы: разрешены include guards, платформенные #if и compile-time флаги; функциональные макросы запрещены
- Ссылки и указатели в параметрах функций: решение по месту (вариант C), в соответствии с ситуацией
- assert: обязателен расширенно для инвариантов и невозможных состояний в src_base; в платформенных адаптерах использовать для проверок предпосылок
- Логирование: обязательно для ошибок I/O и ошибок платформенных API в src_cons/src_win/src_andr; использовать уровни Error/Warn/Info (диагностический минимум), в игровом и рендер-цикле без спама каждый кадр

## Workflow Discipline

- Каждая задача должна сохранять working state проекта
- Предпочитать incremental stabilization
- Предпочитать reviewable isolated changes
- Избегать mass rewrites
- Build/test verification обязательны для critical logic
- Предпочитать deterministic behavior и predictable execution
