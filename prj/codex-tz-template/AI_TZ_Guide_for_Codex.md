# Стандарт подготовки ТЗ для Codex через ChatGPT

Файл-проводник

## 0. Назначение документа

Этот документ служит «проводником» между разработчиком, ChatGPT и кодовой моделью (Codex, Copilot, GPT в IDE и т.п.).

Он помогает:

* разработчику — шаг за шагом собрать информацию о задаче и оформить её в виде качественного ТЗ;
* ChatGPT — понять, как вести интервью и как из ответов собрать структурированное ТЗ;
* Codex — получить чёткое, самодостаточное задание, по которому можно уверенно писать и менять код.

Документ можно:

* хранить в репозитории (например, `docs/AI_TZ_Guide_for_Codex.md`);
* загружать в чат с ИИ каждый раз, когда нужно подготовить ТЗ;
* использовать как стандарт внутри команды.

## 1. Инструкция для ИИ (ChatGPT и подобные модели)

Если этот документ загружен в чат, действуй по следующему протоколу:

1. Определи, что пользователь хочет:
   чаще всего он пишет что-то вроде: «Хочу подготовить ТЗ для Codex для задачи X».

2. Перейди в режим «Ассистент по подготовке ТЗ для Codex»:

   * Сначала задай разработчику вопросы из раздела
     **3. Шаг 1. Сбор контекста (интервью-вопросы)**.
   * Если ответы слишком общие — уточняй, но не придумывай важные требования без явного согласия пользователя.
   * Помни: лучше задать 1–2 дополнительных уточняющих вопроса, чем оставить критичные пробелы.

3. После того как блок вопросов пройден:

   * Сформируй черновик ТЗ по структуре из раздела
     **4. Шаг 2. Шаблон структуры ТЗ**.
   * Подставь в шаблон всё, что пользователь рассказал в ответах.

4. Покажи разработчику собранное ТЗ:

   * Отдельно обозначь, что это «Черновик ТЗ для Codex».
   * При необходимости предложи сократить/уплотнить формулировки.

5. Если разработчик просит «сжать ТЗ»:

   * Сохрани структуру, убери повторения.
   * Оставь все ключевые технические требования и ограничения.

6. В конце пройди с разработчиком по чек-листу из раздела
   **6. Шаг 4. Чек-лист перед отправкой ТЗ Codex’у**:

   * Проверь, что всё покрыто.
   * При необходимости добавь/уточни пункты.

7. В итоговом ответе всегда выдай ТЗ **единым цельным блоком**, чтобы разработчик мог его:

   * вставить в файл `CODING_SPEC.md` или подобный;
   * использовать как промпт/комментарий для Codex/IDE.

## 2. Инструкция для разработчика: как пользоваться проводником

1. Загрузите этот файл в чат с ChatGPT (или аналогичной моделью).

2. Напишите в чат примерно так:

   * «Хочу подготовить ТЗ для Codex на задачу: [краткое описание]»
   * или: «Помоги сделать техническое задание для кодовой модели, которая будет …»

3. Ассистент начнёт задавать вам вопросы по разделу
   **3. Шаг 1. Сбор контекста (интервью-вопросы)**.

4. Отвечайте:

   * максимально конкретно;
   * по делу;
   * не стесняйтесь говорить: «ещё не решил» или «это не важно» — ассистент поможет зафиксировать неопределённости.

5. После интервью ассистент:

   * соберёт ТЗ по структурному шаблону;
   * покажет его вам;
   * по вашему запросу — сократит и приведёт к удобному формату.

6. Вы можете:

   * сохранить ТЗ в репозиторий (например, `AI/CODING_SPEC.md`);
   * вставить его в комментарий к файлу, который будет редактировать Codex;
   * использовать как стандартный промпт для плагина ИИ в IDE.

7. Перед использованием ТЗ:

   * пройдите вместе с ассистентом по чек-листу в разделе
     **6. Шаг 4. Чек-лист перед отправкой ТЗ Codex’у**;
   * при необходимости доработайте формулировки.

## 3. Шаг 1. Сбор контекста (интервью-вопросы)

Этот блок — набор вопросов, которые ИИ задаёт разработчику.
Отвечайте на них по порядку. Ассистент может адаптировать формулировки под вашу задачу.

### 3.1. Проект и контекст

1. Что это за проект? (backend, frontend, мобильное приложение, DevOps-скрипт, инфраструктура, монолит/микросервисы и т.п.)
2. На каком языке и фреймворке он написан? (PHP/Laravel, JS/React, Python/FastAPI и т.д.)
3. Есть ли уже существующий код, или это задача «с нуля»?
4. Где лежат ключевые файлы/папки, с которыми будет работать Codex?

   * Примеры: `src/`, `app/`, `config/`, `bitrix/`, `infra/`, `deploy/`.

### 3.2. Цель задачи

1. Как звучит цель задачи в 1–3 предложениях, если объяснять коллеге?
2. К какому типу задач это ближе:

   * добавление новой функциональности,
   * рефакторинг,
   * миграция,
   * настройка окружения/инфраструктуры,
   * автоматизация (скрипты, CI/CD),
   * другое?

### 3.3. Текущая реализация (если есть)

1. Что сейчас уже есть по этой задаче:

   * скрипты,
   * сервисы,
   * модули,
   * конфиги?
2. В чём проблема или ограничение текущего решения?
3. Нужно ли сохранить обратную совместимость? Если да — в чём именно?

### 3.4. Требования к результату

1. Что конкретно должно появиться в итоге:

   * новые скрипты,
   * новые endpoints,
   * новые компоненты,
   * конфиги,
   * инфраструктурные файлы?
2. Какие функции/действия должен выполнять результат?
3. Какие режимы работы должны быть?

   * Например, как в кейсе окружения:
     — установка нового проекта,
     — настройка существующего,
     — режим `clean` для очистки.
4. Какие параметры/настройки должен принимать код?

   * CLI-аргументы, переменные окружения, настройки в конфигурационных файлах.

### 3.5. Ограничения и запреты

1. Какие технологии/библиотеки/подходы **нельзя** использовать?

   * Например: `dnf/yum`, Bitrix-репозитории, ansible, устаревшие API.
2. Какие версии ОС/языка/фреймворка обязательны?

   * Например: Ubuntu 22.04, PHP 8.2, Laravel 10.
3. Есть ли части кода/папки, которые Codex вообще не должен менять?

### 3.6. Архитектура и стиль кода

1. Есть ли уже принятый архитектурный подход?

   * Слои (application/domain/infrastructure), MVC, DDD, чистая архитектура и т.д.
2. Есть ли требования к стилю кода и форматированию?

   * PSR-12 для PHP, ESLint/Prettier для JS, PEP8 для Python и т.д.
3. Нужно ли добавлять документацию (docblocks, PHPDoc, JSDoc, комментарии к публичным методам)?

### 3.7. Входные и выходные данные

1. Какие входные данные ожидает код?

   * HTTP-запросы, CLI-аргументы, файлы, JSON, очереди.
2. Какие выходные данные должен возвращать?

   * HTTP-ответы, JSON, DTO, файлы, сообщения в логах.
3. Есть ли требования к форматам (например, структура JSON, кодировка, схема)?

### 3.8. Тестирование и приёмка

1. Какие виды тестов нужны:

   * unit, integration, e2e?
2. Должен ли Codex создавать/обновлять тесты?
3. Как вы будете проверять, что задача сделана:

   * список шагов руками;
   * успешный запуск команды;
   * прогон тестов;
   * определённый результат в UI/API?

### 3.9. Особенности взаимодействия с Codex

1. В каких файлах Codex должен работать в первую очередь?
2. Какие файлы ему **нельзя трогать** (генерируемые, легаси, сторонние модули)?
3. Нужно ли создавать новые файлы/папки или только править существующие?

## 4. Шаг 2. Шаблон структуры ТЗ

После того как ответы собраны, ИИ собирает их в ТЗ по следующей структуре.

Этот шаблон — каркас.
Ассистент должен подставить в него конкретику из ответов разработчика.

### 4.1. Контекст проекта

* Краткое описание проекта: назначение, стек (языки, фреймворки).
* Важные пути:

  * где лежат ключевые файлы/директории, с которыми нужно работать.

### 4.2. Текущая реализация и проблема

* Что уже существует (скрипты, сервисы, конфиги).
* В чём ограничения/проблемы текущего решения.
* Нужно ли сохранить обратную совместимость и в каком объёме.

### 4.3. Цель задачи

* 1–3 предложения: что именно нужно получить.
* Тип задачи: новая фича / рефакторинг / миграция / инфраструктура / другое.

### 4.4. Требования к функционалу

* Список функций/действий, которые должен выполнять результат.
* Возможные режимы работы (если есть).
* Принимаемые параметры и настройки (CLI, HTTP, конфиги, ENV).

### 4.5. Ограничения и запреты

* Запрещённые технологии/библиотеки/подходы.
* Обязательные версии ОС/языков/фреймворков.
* Части кода/папки, которые нельзя менять.

### 4.6. Требования к архитектуре и коду

* Желаемая структура: функции, классы, модули, слои.
* Требования к читаемости и оформлению (кодстайл, комментарии, документация).
* Особые архитектурные требования: расширяемость, тестируемость, loose coupling.

### 4.7. Формат результата

* Какие файлы нужно создать.
* Какие файлы нужно изменить.
* Примерное содержание ключевых файлов (назначение, основная логика).

### 4.8. Тестирование и критерии приёмки

* Какие тесты должны быть (unit/integration/e2e).
* Какие проверки должен пройти код, чтобы считаться готовым.
* Как и кто будет проверять результат (скрипт, ручные шаги, CI).

### 4.9. Особые указания для Codex

* Где допустимы изменения, а где — нет.
* Нужно ли генерировать документацию/README/комментарии.
* Все ли ограничения нужно соблюдать строго (или есть мягкие пожелания).

## 5. Шаг 3. Пример готового ТЗ (по мотивам реального кейса)

Ниже — пример ТЗ, которое можно использовать как ориентир.
Его можно показывать разработчику, чтобы он понимал, «как должно выглядеть» итоговое ТЗ.

Пример: миграция BitrixEnv → скрипт окружения для Laravel (SimaiEnv).

> В папке `bitrix/` лежит наследие BitrixEnv 9:
> — `bitrix/root/bitrix-env-9.sh`
> — `bitrix/opt/webdir/`
> Эти файлы используются только как референс архитектуры и подходов, их код переносить нельзя.
>
> Цель: создать новый скрипт `simai-env.sh` под Ubuntu 22.04+, который устанавливает и настраивает окружение для Laravel-проектов и экосистемы SIMAI, не завися от BitrixEnv и не используя его пакеты.
>
> Требования к функционалу:
>
> 1. Использовать `apt` и стандартные репозитории Ubuntu (+ PPA `ondrej/php` для PHP).
> 2. Устанавливать:
>    – `nginx`;
>    – `php-fpm 8.2` + расширения (mbstring, intl, curl, zip, xml, gd/imagick, pdo_mysql, opcache);
>    – `mysql-server` или Percona 8.0+;
>    – `redis`;
>    – `nodejs`;
>    – `composer`;
>    – базовые утилиты (git, curl, unzip, htop, rsyslog, logrotate).
> 3. Создавать пользователя `simai` (если не существует) и директорию проекта `/home/simai/www/[project_name]`.
> 4. Поддерживать два режима работы:
>    – режим A: развёртывание нового Laravel-проекта (`composer create-project`, настройка `.env`, `key:generate`, `storage:link`, миграции по флагу);
>    – режим B: настройка существующего Laravel-проекта по параметру `--path` (проверка структуры, `composer install` при необходимости, настройка `.env`, `key:generate`).
> 5. Настраивать nginx-хост:
>    – `root=/home/simai/www/[project_name]/public`;
>    – `index index.php`;
>    – `try_files $uri $uri/ /index.php?$query_string;`;
>    – прокидывать PHP-запросы на сокет/порт php-fpm.
> 6. Создавать cron-задание для `php artisan schedule:run` и systemd unit для `php artisan queue:work`.
> 7. Поддерживать режим `clean`, который удаляет созданные nginx-конфиги, systemd-юнит, cron-задания и, по флагу, директорию проекта и БД.
>
> Ограничения и запреты:
> – запрещено использовать `dnf/yum/rpm`, подключать Bitrix/EPEL/Remi/Percona RPM-репозитории;
> – запрещено использовать ansible;
> – запрещено поднимать Apache (httpd) как основной веб-сервер;
> – запрещено копировать код/конфиги из BitrixEnv дословно (можно использовать только идеи структуры);
> – скрипт должен работать на Ubuntu 22.04+.
>
> Требования к архитектуре:
> – скрипт `simai-env.sh` должен быть разбит на функции:
> `install_php()`, `install_nginx()`, `install_mysql()`, `install_redis()`,
> `configure_project()`, `configure_nginx_host()`, `configure_cron_and_queue()`,
> `clean_environment()`;
> – должна быть общая функция логирования, пишущая в лог-файл;
> – параметры CLI должны разбираться через стандартный `getopts` или аналог.
>
> Формат результата:
> – создать файл `simai-env.sh` в корне проекта;
> – создать шаблон nginx-конфига `templates/nginx-laravel.conf`;
> – создать systemd unit `systemd/laravel-queue.service`;
> – создать или обновить `README.md` с инструкцией по установке и использованию.
>
> Критерии приёмки:
> – запуск `simai-env.sh` на чистой Ubuntu 22.04 в режиме установки нового проекта приводит к рабочему Laravel-сайту, доступному по указанному домену;
> – скрипт не использует Bitrix-зависимости и не обращается к Bitrix-репозиториям;
> – после запуска режима `clean` окружение, созданное скриптом (сайты, юниты, cron), корректно удаляется.

## 6. Шаг 4. Чек-лист перед отправкой ТЗ Codex’у

Перед тем как копировать ТЗ в Codex/IDE, разработчик проходит по чек-листу:

1. Есть ли чётко сформулированная цель задачи (1–3 предложения)?
2. Описан ли контекст проекта и стек (язык, фреймворк, ОС)?
3. Указаны ли пути к ключевым файлам/папкам, с которыми будет работать Codex?
4. Есть ли список функциональных требований (что код обязан делать)?
5. Есть ли список ограничений и запретов (что нельзя использовать/менять)?
6. Прописан ли формат результата:

   * какие файлы создать,
   * какие файлы изменить?
7. Описаны ли критерии приёмки (как вы поймёте, что всё работает)?
8. Нет ли ссылок на «см. выше/как обсуждали ранее» или внешние документы без краткого пересказа?
9. Можно ли это ТЗ **целиком скопировать одним блоком** в IDE/плагин?
10. Нет ли явных противоречий (например, требование «не использовать Laravel» и одновременно «создать Laravel-контроллер»)?

Если хотя бы на один вопрос ответ «нет» — стоит доработать ТЗ вместе с ассистентом.

## 7. Дополнительные подсказки для разных типов задач (опционально)

Этот раздел можно расширять внутри команды по мере накопления опыта.

### 7.1. Backend API (REST/GraphQL)

Рекомендуется дополнительно указать:

* форматы входных/выходных JSON;
* HTTP-статусы и обработку ошибок;
* правила пагинации/фильтрации;
* требования к авторизации/аутентификации.

### 7.2. Frontend (SPA/компоненты)

Рекомендуется дополнительно указать:

* дизайн-гайд (UI-kit, фреймворк: Bootstrap/Tailwind/свой);
* требования к адаптивности и доступности (a11y);
* целевые браузеры / минимальные версии.

### 7.3. DevOps/окружение/деплой

Рекомендуется дополнительно указать:

* целевые ОС и версии;
* package-manager (apt/yum/pacman и т.д.);
* требования к idempotency (повторный запуск скрипта);
* требования к безопасности (пользователи, права, порты, firewall).
